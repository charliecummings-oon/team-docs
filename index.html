Claude Code: How It All Fits Together

Interactive guide for first-time users | Click any card to expand details

Claude Code reads instructions from three tiers, loaded in order. Global rules apply everywhere. Project rules add repo-specific conventions. Settings configure hooks and integrations.

Tier 1 — Global (all sessions, all projects)
~/.claude/CLAUDE.md click to expand
Non-negotiable rules for every session

What to put here: Anything you’d tell a new team member on day one that applies regardless of project.

Examples from our setup:

  • After any code changes → update changelog + business context files
  • After EVERY conversation → update business context log
  • Before writing any database query → read the query reference guide
  • All data analysis must be script-driven (never mental math)
Why mandatory? Claude has predictable failure modes: forgetting to maintain context, doing approximate math. Making rules explicit and global prevents drift.

Tier 3 — Settings (hooks + integrations)
~/.claude/settings.json click to expand
SessionStart hook and MCP server connections
{
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "bash ~/.claude/scripts/session-context.sh"
      }]
    }]
  }
}

This is where you define your SessionStart hook (fires every session) and any MCP server connections for external APIs.

The memory directory (~/.claude/projects/<path>/memory/) persists across sessions. These files are how Claude accumulates knowledge over time.

MEMORY.md — The Index click to expand
First 200 lines injected into every session’s system prompt automatically

Use as a concise index:

  • Project overview (what you’re building, what stack)
  • Key metrics or goals Claude should always consider
  • Links to other memory files with one-line descriptions

Keep it tight. The 200-line limit is real. Detailed material goes in dedicated files.

CURRENT_STATE.md — Where You Are Right Now click to expand
Curated strategic summary (~50-80 lines), rewritten when the landscape shifts. Injected in full every session.

Unlike the append-only logs below, this file gets rewritten (not appended) when the situation changes meaningfully.

BUSINESS_CONTEXT.md expand
What you know (insights, decisions)

Updated after every conversation — not just code sessions. Captures:

  • Stakeholder feedback from meetings
  • Data findings and implications
  • Strategic decisions and reasoning
  • Domain term definitions

Latest entry injected by the hook. When the file exceeds ~800 lines, entries older than 14 days get moved to BUSINESS_CONTEXT_ARCHIVE.md. Before archiving, verify active context is captured in CURRENT_STATE.md.

How these connect MEMORY.md is the always-on index. CURRENT_STATE.md provides the full strategic picture. CHANGELOG and BUSINESS_CONTEXT get their latest entries injected by the SessionStart hook. Domain reference files are read on demand. Together, they give Claude both broad awareness and deep domain knowledge.

Every session follows this lifecycle. The SessionStart hook provides automatic continuity — no manual context loading needed.

Session Start click to expand
You open Claude Code in a project directory

Three things happen automatically:

  1. Global ~/.claude/CLAUDE.md loaded (your mandatory rules)
  2. Project CLAUDE.md loaded (repo conventions + domain)
  3. SessionStart hook fires
Hook Fires click to expand
Current state + latest changelog + latest business context injected
#!/bin/bash
MEMORY_DIR="~/.claude/projects/<your-project>/memory"

echo "=== CURRENT STATE ==="
cat "$MEMORY_DIR/CURRENT_STATE.md"

echo "=== RECENT CHANGELOG ==="
awk '/^## [0-9]{4}-/{if(found) exit; found=1} found' \
  "$MEMORY_DIR/CHANGELOG.md"

echo "=== RECENT BUSINESS CONTEXT ==="
awk '/^## [0-9]{4}-/{if(found) exit; found=1} found' \
  "$MEMORY_DIR/BUSINESS_CONTEXT.md"

Injects three things: (1) CURRENT_STATE in full (curated strategic summary), (2) latest dated section from CHANGELOG, (3) latest dated section from BUSINESS_CONTEXT.

Next Session — Automatic Continuity click to expand
Hook injects updates from the previous session

The feedback loop:

  1. Monday: work session → memory files updated
  2. Tuesday: new session → hook injects Monday’s updates → Claude picks up where you left off
  3. Wednesday: boss gives feedback → business context updated
  4. Thursday: new session → Claude knows what the boss said

No “where were we?” — the system remembers for you.

Two categories of guardrails have been the most impactful: forcing analytical rigor and verifying document accuracy.

Analytical Rigor

“All data analysis must use Python scripts to compute results. Every number presented must come from script output. Never do mental math, eyeball data, or approximate.”

Claude will do mental math if you let it. It’ll eyeball a CSV and say “about 40% are profitable” when the real number is 34%. Put this rule in your global CLAUDE.md to eliminate it.

Artifact Verification

If Claude produces documents with data claims, consider building a verification pipeline. This is a set of scripts that work together to catch errors before stakeholders do.

1
Analysis Script (--claims flag)
Your main analysis script does double duty. Run normally: human-readable report. Run with --claims: also outputs a JSON manifest of every verifiable fact (values, types, tolerances). This is the source of truth.
↓
2
Verification Script
Takes an HTML artifact + the claims JSON. Runs two kinds of checks:
Deterministic (PASS/FAIL): Rounded millions, exact integers, percentages, dollar amounts, decimal ratios.
Semantic (WARN): Absolutist claims (“all”, “every”), equivalence claims (“same”, “identical”), causal claims (“because”), status claims (“done”, “complete”).
↓
3
Validation Scripts
Cross-reference your analysis findings against external data sources (e.g., per-message API data) to detect aggregation artifacts, inflated estimates, or methodology issues. These catch problems in the analysis itself, not just the document.
FAIL? Fix the document, re-run. Repeat until 0 FAIL, all WARN addressed.

Put the verification protocol in your project CLAUDE.md so Claude runs it automatically after any document edit.

Real catches This pattern has caught: a claim that said “same creative” when it was actually “same offer” (subject lines differed), a claim of “all 10” that was actually 9 of 10, rounding discrepancies in percentages, and $248K in inflated opportunity estimates from data aggregation artifacts.
