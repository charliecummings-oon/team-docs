How I Use Claude Code

A practical guide for first-time users | February 2026
Covers the principles and patterns that make Claude Code effective for sustained, multi-session work.

1. The Problem: Session Amnesia

Claude Code is a CLI agent that reads files, writes code, runs scripts, and executes terminal commands. Out of the box, every session starts from zero — no memory of past conversations, no knowledge of your codebase conventions, no business context.

That’s fine for one-off tasks. It’s a problem when you’re doing sustained work across days or weeks. You don’t want to re-explain your project’s architecture, your database schema, or what your boss said in yesterday’s meeting.

We solve this with three mechanisms:

  1. Configuration files — written instructions Claude reads at the start of every session (like an employee handbook)
  2. Memory files — a living knowledge base that grows across sessions (like a shared team wiki)
  3. A SessionStart hook — a script that automatically injects the latest context into every session (like a daily briefing)
The result Every new session starts with full awareness of your project, conventions, recent work, and strategic context. No manual re-explanation needed.

2. Configuration Hierarchy

Claude Code reads instructions from three tiers, loaded in order. Think .gitconfig (global) vs .git/config (repo-level).

Tier 1: Global — ~/.claude/CLAUDE.md

Rules that apply to every project, every session. This is where you put non-negotiable behaviors. Examples from our setup:

  • After any code changes → update the changelog and business context files
  • After EVERY conversation → update the business context log
  • Before writing any database query → read the query reference guide
  • All data analysis must be script-driven (never mental math)

These exist because Claude has predictable failure modes — forgetting to maintain context, doing approximate math. Making rules explicit and global prevents drift.

What to put here: Anything you’d tell a new team member on day one that applies regardless of which project they’re working on.

Tier 2: Project — <repo>/CLAUDE.md

Each repository gets its own CLAUDE.md at the root with project-specific context:

What to IncludeWhy
What the project doesEnough that Claude can reason about it without asking
Code conventionsFormatter, line length, naming patterns, preferred libraries
Domain conceptsYour framework, your terminology, your business logic
File organizationWhere scripts live, where docs go, how things are structured
Key workflowsHow to run tests, how to deploy, how to verify output
Key principle If you find yourself re-explaining something every session, it belongs in the CLAUDE.md. Put enough domain context in there that Claude can work autonomously without you narrating every step.

Tier 3: Settings — ~/.claude/settings.json

Configuration for hooks and external integrations:

{ "hooks": { "SessionStart": [{ "hooks": [{ "type": "command", "command": "bash ~/.claude/scripts/session-context.sh" }] }] } }

This is where you define your SessionStart hook (see Section 4) and any MCP server connections for external APIs.

3. The Memory System

Claude Code has a built-in memory directory at ~/.claude/projects/<project-path>/memory/. Files here persist across sessions. The key is how you organize them.

MEMORY.md — The Index

The first 200 lines of this file get injected into every session’s system prompt automatically. Use it as a concise index:

  • Project overview (what you’re building, what stack)
  • Key metrics or goals Claude should always consider
  • Links to other memory files (with one-line descriptions of each)

Keep it tight. The 200-line limit is real. Detailed reference material goes in dedicated files, not here.

CURRENT_STATE.md — Where You Are Right Now

A curated summary (~50-80 lines) of the current strategic landscape. Unlike the append-only logs below, this file gets rewritten (not appended) when the situation shifts meaningfully.

SectionWhat It Contains
Active projectsWhat’s in flight, what phase each is in, what’s blocked
Key findingsConclusions still informing current work
Open decisionsUnresolved questions awaiting input or analysis
Stakeholder contextWhat leadership has said that’s still directing work

Gets injected in full into every session by the hook (see Section 4). Rewrite when the landscape shifts: new key finding, stakeholder directive, project phase change, or major decision.

CHANGELOG.md — What Changed

A running log of every code and config change. Format with dated sections:

## 2026-02-18 ### [project-name] — Brief description What was done and why. **Files created/modified:** - path/to/file.py — What it does

Gets updated automatically by Claude (enforced by global CLAUDE.md rules). The latest entry gets injected into every new session by the hook.

BUSINESS_CONTEXT.md — What You Know

A running log of business insights, strategic decisions, meeting notes, and observations. Updated after every conversation — not just code sessions.

This is your institutional memory. It captures things like:

  • Stakeholder feedback from meetings
  • Data findings and their implications
  • Strategic decisions and the reasoning behind them
  • Definitions and clarifications of domain terms

Archive rotation: When this file exceeds ~800 lines, move entries older than 14 days to BUSINESS_CONTEXT_ARCHIVE.md. Before archiving, verify active context is captured in CURRENT_STATE.md.

Domain Reference Files

Create dedicated .md files for any knowledge Claude needs to consult regularly:

ExampleWhat It Contains
Database query guideSchemas, default filters, connection config, gotchas
Writing playbookHow to structure briefs, proposals, or reports for your team
Stakeholder frameworkLeadership vision, open questions, exploration priorities

Claude reads these on demand. A rule like “before writing any SQL, read the query guide” (in global CLAUDE.md) ensures it actually does.

4. The SessionStart Hook

This is what makes the memory system proactive instead of passive. It’s a short shell script that runs at the start of every session:

#!/bin/bash MEMORY_DIR="~/.claude/projects/<your-project>/memory" echo "=== CURRENT STATE ===" cat "$MEMORY_DIR/CURRENT_STATE.md" echo "" echo "=== RECENT CHANGELOG (last entry) ===" awk '/^## [0-9]{4}-/{if(found) exit; found=1} found' \ "$MEMORY_DIR/CHANGELOG.md" echo "" echo "=== RECENT BUSINESS CONTEXT (last entry) ===" awk '/^## [0-9]{4}-/{if(found) exit; found=1} found' \ "$MEMORY_DIR/BUSINESS_CONTEXT.md"

What it does: Injects three things into every session: (1) CURRENT_STATE in full (curated strategic summary), (2) latest dated section from CHANGELOG, (3) latest dated section from BUSINESS_CONTEXT.

The feedback loop
  1. You have a conversation → Claude updates logs (and rewrites CURRENT_STATE if the landscape shifted)
  2. You close the session
  3. Next time you open Claude Code → hook injects everything
  4. Claude picks up with full awareness → you continue where you left off
No “where were we?” needed. Session continuity is automatic.

5. Guardrails That Matter

Analytical Rigor

If you use Claude Code for any kind of data analysis, put this in your global CLAUDE.md:

All data analysis must use Python scripts (pandas, numpy, etc.) to compute results. Every number presented must come from script output. Never do mental math, eyeball data, or approximate — even on small datasets.

Claude will do mental math if you let it. It’ll eyeball a CSV and say “about 40% are profitable” when the real number is 34%. Making the rule explicit eliminates this.

Artifact Verification

If Claude produces documents with data claims (reports, presentations, briefs), consider building a verification pipeline. This is a set of scripts that work together to catch errors before stakeholders do.

The pipeline has three roles:

  1. Analysis script (--claims flag) — Your main analysis script does double duty. Run normally: human-readable report. Run with --claims: also outputs a JSON manifest of every verifiable fact (values, types, tolerances). This is the source of truth.
  2. Verification script — Takes an HTML artifact + the claims JSON. Runs deterministic checks (PASS/FAIL: rounded millions, exact integers, percentages, dollar amounts) and semantic checks (WARN: absolutist claims, equivalence claims, causal claims, status claims).
  3. Validation scripts — Cross-reference your findings against external data sources (e.g., per-message API data) to detect aggregation artifacts, inflated estimates, or methodology issues. These catch problems in the analysis itself, not just the document.

Put the verification protocol in your project CLAUDE.md so Claude runs it automatically after any document edit. The protocol: run analysis with --claims → run verification → fix all FAILs → review all WARNs → re-run clean.

This pattern has caught: a claim that said “same creative” when it was actually “same offer,” a claim of “all 10” that was actually 9 of 10, rounding discrepancies in percentages, and $248K in inflated opportunity estimates from data aggregation artifacts.

6. Extending Claude Code

Google Drive Access

We built a small Python utility that lets Claude search, read, and download files from Google Drive:

python3 gdrive_auth.py search "query" # Search files python3 gdrive_auth.py read FILE_ID # Read a Google Doc as text python3 gdrive_auth.py download FILE_ID # Download a file python3 gdrive_auth.py list # List recent files

Useful when your team’s documentation lives in Drive. Claude can pull in context from shared docs without you copy-pasting.

The Pattern for Any External Tool

The general approach:

  1. Write a small CLI utility that handles auth and API calls
  2. Document the commands in your MEMORY.md or a reference file
  3. Claude will use it like any other command-line tool

You can also use MCP servers (Model Context Protocol) to give Claude direct API access to external services. These are configured in settings.json and give Claude native tool access to third-party platforms without custom scripts.

7. Day-to-Day Patterns

Starting a new session

Open Claude Code in your repo directory. The hook fires, context loads, Claude is up to speed. Just start working.

Cross-session continuity

Monday you do deep analysis. Tuesday you open a new session — the hook injects Monday’s changelog and business context. Claude picks up where you left off. Wednesday your boss gives feedback — you tell Claude, it updates the business context. Thursday’s session knows what the boss said.

Adding domain knowledge

What You LearnedWhere It Goes
New database table or columnYour query reference guide
New strategic decisionCaptured in BUSINESS_CONTEXT automatically
New convention or processRelevant CLAUDE.md (global or project)
New reference documentCreate file in memory directory, add to MEMORY.md index

Building documents for stakeholders

Claude writes HTML with inline styles (no external dependencies) → you copy-paste into Google Docs with formatting intact. If you have a verification pipeline, run it before sharing.
